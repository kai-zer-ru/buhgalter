# Списание ретроактивных платежей при создании кредита

Реализовано в **v1.1.0**. Исходный черновик ТЗ ниже.

Сейчас при создании кредита с флагом **«Добавлен в учёт после выдачи»** (`added_retroactively`) все платежи с датой **до сегодня** попадают в график как `kind=retroactive`: считаются уже погашенными для `paid_amount`, **не создают операций** на счёте и **исключаются из статистики** (`exclude_from_stats=1`). Чекбокс «Создать будущие операции» при этом скрыт.

Частый сценарий: кредит выдали месяц назад, три платежа уже прошли — два оплачены «где-то вне приложения», а **последний** реально списался с дебетового счёта. Нужно учесть кредит задним числом **и** отразить одно (или несколько) фактических списаний.

## Зачем

| Сейчас | Нужно |
|--------|--------|
| Все прошлые платежи — только метка «Учтён при добавлении» | Часть прошлых платежей можно пометить **«Списан со счёта»** |
| Баланс счёта не меняется | Для помеченных — расход на `debit_account_id` в дату платежа |
| В статистике прошлые платежи не видны | Помеченные списания — обычный расход (категория «Кредиты») |

Без этой опции пользователь вынужден вручную заводить операции и сверять `paid_amount` с графиком.

## Область действия

- Только **форма создания** кредита (`CreditForm`), при включённом `added_retroactively`
- Только строки графика со статусом **«Учтён при добавлении»** (дата платежа &lt; начала текущих суток в timezone пользователя)
- После сохранения кредита — **не менять** через эту опцию (редактирование графика v1.1 — только суммы у `scheduled`); досписание ретро — отдельная задача, вне scope

## Поведение в UI

### Чекбокс «Списан со счёта»

В таблице/карточках графика у каждой ретроактивной строки — опциональный чекбокс (или переключатель) **«Списан со счёта»** рядом с бейджем «Учтён при добавлении».

При сохранении кредита для отмеченных строк:

1. Создаётся операция `expense` на `debit_account_id` (системная категория «Кредиты»), дата = дата платежа, `affects_balance = true`
2. Строка `credit_payments` связывается с `transaction_id`
3. `paid_amount` кредита увеличивается на сумму платежа (как у обычного погашения)
4. `exclude_from_stats = 0` — платёж виден в статистике как расход

Непомеченные ретроактивные строки — как сейчас: без операции, `exclude_from_stats = 1`.

### Правило «активного» платежа

Можно отметить **несколько** прошлых платежей как списанные со счёта, но **редактировать** эту опцию в любой момент можно только у **одной** строки — у **последнего уже отмеченного** (по порядку в графике, от старых к новым).

**Определения:**

- **Ретроактивная строка** — дата &lt; сегодня при включённом `retroactive`
- **Отмеченная** — чекбокс «Списан со счёта» включён
- **Активная для опции** — последняя отмеченная строка в порядке следования в графике; только у неё чекбокс интерактивен для **снятия** отметки; у следующей неотмеченной ретроактивной строки (если есть) — интерактивен для **установки** отметки

**Пример** (платежи 1–4 в прошлом, 5–6 в будущем):

| # | Дата | Статус | Списан со счёта | Активен чекбокс? |
|---|------|--------|-----------------|------------------|
| 1 | янв | Учтён | ✓ | нет (заблокирован) |
| 2 | фев | Учтён | ✓ | нет |
| 3 | мар | Учтён | ✓ | **да** (последний отмеченный) |
| 4 | апр | Учтён | — | да (можно отметить — станет новым последним) |
| 5 | май | Ожидает | — | нет |
| 6 | июн | Ожидает | — | нет |

**Если активный платёж «поменяли»** (сняли галочку, сдвинули дату в будущее, удалили строку в ручном графике, изменили сумму — на усмотрение MVP: достаточно снятия галочки и смены даты):

- Сняли отметку с #3 → активной становится #2 (предыдущая отмеченная); у #3 галочку снова поставить нельзя, пока не снимут #2
- Сдвинули дату #3 в будущее → #3 перестаёт быть ретроактивной, отметка сбрасывается; активной становится #2
- Отметили #4 → #4 становится последним отмеченным и единственной активной для снятия

**Инвариант:** отмеченные строки всегда образуют **непрерывный префикс** с начала ретроактивного блока (нельзя отметить 1 и 3, пропустив 2). При попытке отметить «через один» — автоматически отмечаются все предыдущие ретроактивные строки **или** UI не даёт отметить (предпочтительно: **только непрерывный хвост** от последней ретроактивной назад).

Уточнённое правило хвоста:

- Отмеченные платежи — всегда **последние N** ретроактивных строк подряд (N ≥ 0)
- Пользователь увеличивает N, отмечая следующую ретроактивную снизу вверх
- Уменьшает N, снимая галочку только с **нижней** границы (последней отмеченной)

Так «активный» — всегда нижняя граница хвоста: либо снять отметку с неё, либо расширить хвост на одну строку выше (если ещё не отмечена).

## Модель данных и API (черновик)

Расширение `schedule_seed` при создании:

```json
{
  "schedule_seed": [
    {
      "payment_date": "2025-01-15T00:00:00Z",
      "amount": "14600.00",
      "debit_from_account": true
    }
  ]
}
```

Сервер при `Create`:

- Валидирует: `debit_from_account` только для дат &lt; `todayStart`; отмеченные индексы — непрерывный хвост ретроактивных записей в порядке дат
- Для `debit_from_account=true`: `kind=retroactive` **или** новый подтип / флаг на платеже — на обсуждение; минимум: `retroactive` + `transaction_id` + `exclude_from_stats=0`
- Сумма отмеченных входит в `paid_amount`; неотмеченные ретро — в `paid_amount` как сейчас, без транзакции

Альтернатива без флага в seed: поле `retroactive_debit_count: 3` — «списать со счёта последние 3 ретроактивных платежа» (проще валидировать на бэкенде, совпадает с правилом хвоста).

## Связь с другими опциями

- **`create_transactions`** — по-прежнему только для **будущих** `scheduled` (не ретро)
- **`debit_account_id`** — один счёт и для будущих операций, и для ретро-списаний
- **Ручной график** (`payment_interval=manual`) — то же правило хвоста по порядку строк

## Этапы внедрения

1. **MVP:** `retroactive_debit_count` или флаги в seed; бэкенд + чекбокс-хвост в UI; тесты на `paid_amount` и баланс счёта
2. **UI:** подсказка у чекбокса; мобильные карточки графика
3. **OpenAPI** + i18n

## Открытые вопросы

- [ ] Оставлять `kind=retroactive` при наличии транзакции или вводить `retroactive_debited` / флаг на платеже
- [ ] Меняется ли сумма ретро-строки после отметки — пересчитывать ли уже «замороженные» выше по хвосту
- [ ] Несколько счётов для ретро vs один `debit_account_id`
- [ ] Показывать ли в детальной карточке кредита, какие ретро-платежи были списаны со счёта при создании (read-only)

## Зависимости

- Создание кредита и ретроактивный график (этап 5)
- Системная категория «Кредиты», операции expense
- [Ипотека](mortgage.md) — первоначальный взнос при создании; похожая механика списания, но другая сущность

## Не в scope

- Списание ретро-платежей **после** создания кредита
- Частичное списание (сумма меньше платежа)
- Откат списания при удалении кредита — как у обычных оплаченных платежей (существующая логика)
