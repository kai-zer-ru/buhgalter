# Сброс и восстановление пароля

Не входит в v1. Черновик для обсуждения.

## Зачем

Сейчас сменить пароль можно только в настройках, зная текущий ([`changePassword`](../web/src/lib/api/client.ts)). Админ создаёт пользователей с паролем, но **не может сбросить** пароль существующему. На странице входа нет «Забыли пароль?» — при утере пароля остаётся только ручное вмешательство в БД.

## Направления

### Страница входа

- Ссылка «Забыли пароль?» → форма восстановления (логин)
- Механизм сброса — зависит от канала доставки (см. ниже)
- Одноразовый токен с ограниченным сроком жизни; после смены — инвалидация сессий пользователя

### Админка (`/admin/users`)

- Действие «Сбросить пароль» для выбранного пользователя
- Варианты UX: задать новый пароль вручную **или** сгенерировать временный и показать админу один раз (как при создании API-токена)
- Запись в audit log (`user.password.reset` / `admin.user.password.reset`)

### Канал доставки (self-hosted)

Для локального/самохостного развёртывания e-mail может быть не настроен. Возможные варианты:

- **E-mail** — если в системе появится SMTP (опционально)
- **Telegram / MAX** — ссылка или код сброса в уже подключённый канал уведомлений
- **Только админ** — пользователь обращается к администратору; self-service недоступен без канала

## Зависимости

- Политика паролей и подтверждение — уже есть ([authentication.md](../docs/api/authentication.md))
- Audit log — уже есть
- Уведомления Telegram/MAX — [OpenAPI `/user/notifications`](../docs/api/openapi.yaml) (опциональный канал для self-service)

## Не в scope первой итерации

- Восстановление по секретным вопросам
- 2FA / recovery codes
