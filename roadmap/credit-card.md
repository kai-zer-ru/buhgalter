# Кредитная карта (тип счёта)

Не входит в v1. Черновик для обсуждения — не детальное ТЗ.

Сейчас в v1: счета только **`cash`** и **`bank`** (дебетовые). Модуль **«Кредиты»** ([stage_05](../stage_05_credits.md)) — **целевые займы** с фиксированным графиком (ипотека, потребкредит, рассрочка магазина). **Кредитная карта** — другая сущность: **возобновляемая** задолженность перед банком, лимит, расчётный период, льготный период, минимальный платёж. Смешивать с таблицей `credits` не стоит.

## Зачем отдельный тип

| | Дебетовый (`bank`) | Кредит (модуль) | Кредитная карта |
|---|-------------------|-----------------|-----------------|
| Баланс | деньги **на** счёте | не счёт; списание с дебета | деньги **должны банку** |
| Лимит | — | сумма займа | кредитный лимит |
| Операции | расход уменьшает баланс | платежи по графику | покупка **увеличивает** долг |
| График | — | аннуитет / рассрочка | цикл выписок, не фикс. срок |
| Погашение | — | с дебетового счёта по датам | перевод с дебета **на карту** |

Типичный сценарий: зарплата на дебетовый «Тинькофф», покупки с «Тинькофф Black» (кредитка), раз в месяц перевод с дебета на карту до даты платежа.

## Модель данных (черновик)

**Вариант A (предпочтительный):** `accounts.type = 'credit_card'`, доп. поля на счёте или таблица `credit_card_details` (1:1):

- `bank_id` — как у банковского счёта (иконка банка)
- `credit_limit` — лимит (копейки)
- `billing_day` — день закрытия расчётного периода (1–28)
- `payment_due_day` или `days_after_billing` — когда платить / сколько дней льготного периода
- `interest_rate_annual` — ставка на остаток после льготного периода (опционально, 0 = не считаем проценты в приложении)
- `statement_balance` / расчёт из операций — на обсуждение

**Баланс счёта:**

- семантика **отрицательного** баланса: `-15 000 ₽` = долг 15 000; **или**
- отдельно `owed_amount` + в UI «15 000 из 100 000» (использовано / лимит), а `balance` всегда ≥ 0 для единообразия API

Нужно выбрать одну модель до реализации; от этого зависят переводы, главная и статистика.

**Операции:**

- **Расход на карте** — обычная операция `expense` с `account_id` = карта; увеличивает долг
- **Погашение** — **перевод** `debit_account → credit_card_account`; уменьшает долг
- Превышение лимита — мягкое предупреждение vs hard block (на обсуждение)
- Доход на кредитку (возврат в магазине) — `income` на счёт карты

**Не дублировать:** автосписание минимального платежа — не тот же cron, что у `credits`; логика «минимальный платёж / полное погашение выписки» — отдельная.

## UI и главная

- Создание счёта: тип **«Кредитная карта»**, банк, лимит, дни цикла (MVP: только лимит + начальный долг)
- Иконка / бейдж отличать от дебетового счёта того же банка
- Главная: блок «долг по картам» или карты в списке счетов с красным долгом
- **Двойной баланс** (текущий / с учётом future за месяц) — распространить на карты
- Перевод на карту — подсказка «погашение задолженности»

## Статистика и отчёты

- Расходы по категориям — **в дату покупки**, даже если выписка не закрыта
- Отдельный срез «задолженность по кредитным картам» vs «кредиты» (займы)
- Импорт Cubux: отдельный тип в маппинге счетов

## Уведомления

- Напоминание: «до оплаты выписки N дней», «минимальный платёж X ₽»
- Не путать с триггером `credit_payment` для займов ([stage_08](../stage_08_notifications.md))

## Этапы внедрения

1. **MVP:** тип `credit_card`, лимит, долг, расход и перевод-погашение, главная
2. **Цикл:** billing_day, сумма к оплате по выписке
3. **Проценты и мин. платёж** — опционально
4. **Импорт и bank sync** — [bank-sync.md](bank-sync.md)
