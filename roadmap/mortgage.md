# Ипотека (тип кредита)

Не входит в v1.1. Черновик для обсуждения — не детальное ТЗ.

Сейчас модуль **«Кредиты»** покрывает потребительские займы и рассрочки: одна сумма `principal_amount`, аннуитет или равные платежи без процентов, график по интервалу (месяц / неделя / две недели / вручную). С v1.1 — банковский аннуитет с «коротким» последним платежом и ручное редактирование сумм неоплаченных строк графика.

**Ипотека** — отдельный **тип кредита** внутри того же модуля: та же таблица `credits`, те же платежи и автосписание, но другая модель ввода (цена объекта, первоначальный взнос) и уточнённый расчёт процентов.

## Зачем отдельный тип

| | Потребительский кредит (сейчас) | Ипотека |
|---|--------------------------------|---------|
| Сумма займа | вводится сразу как `principal_amount` | цена объекта минус **первоначальный взнос** (или ввод суммы кредита отдельно) |
| Первоначальный взнос | нет | есть; опционально списание со счёта при создании |
| Тело долга в графике | вся `principal_amount` | только финансируемая часть (без взноса) |
| Проценты | упрощённая ставка за период (`ставка / 12` для месяца) | ближе к банку: проценты на остаток с учётом **фактических дней** между датами платежей |
| Типичный срок | месяцы / годы | 10–30 лет, только месячный интервал (MVP) |
| Смена типа | — | **один раз** для уже сохранённого кредита (см. ниже) |

Потребкредит и ипотеку не стоит смешивать в одной форме без явного типа: иначе пользователь вводит «сумму кредита», а для ипотеки нужны «цена квартиры» и «взнос».

## Модель данных (черновик)

Расширение таблицы `credits` (имена полей — на обсуждение):

```sql
-- credit_kind TEXT NOT NULL DEFAULT 'consumer'
--   CHECK (credit_kind IN ('consumer', 'mortgage'))
-- property_price   INTEGER  -- цена объекта, копейки (только mortgage)
-- down_payment     INTEGER  -- первоначальный взнос, копейки (только mortgage)
-- down_payment_account_id TEXT REFERENCES accounts(id)  -- если взнос списан со счёта
-- down_payment_transaction_id TEXT REFERENCES transactions(id)  -- связанная операция
-- kind_locked_at TEXT  -- после сохранения: тип нельзя менять
```

**Семантика сумм:**

- `property_price` — полная стоимость (для справки в UI и отчётах)
- `down_payment` — взнос; `0` допустим
- **`principal_amount`** (как сейчас) = `property_price - down_payment` — то, на что строится график и считается `paid_amount` / остаток
- Альтернатива MVP: не хранить `property_price`, а только `down_payment` + `principal_amount`; цена выводится как сумма

**Первоначальный взнос и счета:**

1. **Только учёт** — взнос уменьшает финансируемую сумму, на баланс счетов не влияет (уже «вне приложения» оплачен)
2. **Списание со счёта** — при создании кредита: расход (или перевод) с выбранного `debit_account_id` / отдельного счёта на сумму взноса; в статистике — отдельная операция, не платёж по графику

Вариант 2 — опциональный чекбокс в форме: «Списать взнос со счёта сейчас».

## Расчёт графика и процентов

Текущий генератор (`schedule.go`): для каждого периода процент = `остаток × (годовая_ставка / 12 / 100)` с усечением копеек; платежи 1…N−1 равны, последний закрывает остаток.

Для ипотеки целевое поведение:

- Проценты за интервал: `остаток × ставка × (дней_в_периоде / 365)` (или `/ 366` в високосном году — уточнить по банку-эталону)
- Даты платежей — как сейчас (`issue_date` + интервал), для MVP ипотеки достаточно `payment_interval = 'month'`
- Последний платёж — остаток тела + проценты за последний период (как в v1.1 для потребкредита)
- Превью графика (`POST /credits/schedule/preview`) должно использовать тот же алгоритм, что и сохранение

**Открытый вопрос:** насколько точно повторять конкретный банк (Альфа, Сбер, …). Для roadmap достаточно «ближе к выписке, чем упрощённый месячный коэффициент»; эталонные тесты — по PDF/выписке пользователя.

## Смена типа кредита (один раз)

Требование из roadmap:

- **До первого сохранения** (создание): переключатель «Потребительский» / «Ипотека» — сколько угодно раз, поля и превью графика пересчитываются
- **После сохранения:** тип **фиксируется** (`kind_locked_at` или флаг `credit_kind_locked`)
- **Исключение — однократная конвертация** уже существующего кредита:
  - Действие в UI: «Сменить тип на ипотеку» / «Сменить на потребительский» (если данные совместимы)
  - Доступно **ровно один раз** на кредит; после конвертации снова блокировка
  - При смене consumer → mortgage: запросить недостающие поля (взнос, цена); пересчитать график **только для неоплаченных** `scheduled` или предупредить о ручной проверке — на обсуждение
  - При смене mortgage → consumer: потеря полей `property_price` / `down_payment` в UI (данные можно оставить в БД, но не показывать)

Конвертация не должна ломать уже оплаченные платежи и `paid_amount`.

## UI

### Создание / редактирование

- Тип кредита: сегмент или select в начале формы (`CreditForm`)
- **Ипотека:** поля «Цена объекта», «Первоначальный взнос», read-only «Сумма кредита» (= разница); чекбокс списания взноса
- **Потребительский:** как сейчас — сразу сумма кредита
- Превью графика и аннуитетный платёж — с учётом типа
- Бейдж на карточке кредита: «Ипотека» (аналог `is_installment` для рассрочки)

### Карточка кредита

- Показывать цену объекта, взнос, сумму кредита (для mortgage)
- Редактирование графика (v1.1) — без изменений по scope
- Кнопка однократной смены типа — только если ещё не использована и кредит `active`

## API (черновик)

- `credit_kind` в `CreateCreditRequest` / `Credit` response
- Поля `property_price`, `down_payment`, `down_payment_debit_account_id` (optional)
- `PATCH /credits/{id}/kind` — однократная смена типа с телом новых полей
- Валидация: `down_payment < property_price`, суммы > 0, для mortgage при `interest_rate = 0` — либо запрет, либо трактовка как «беспроцентная ипотека» (маловероятно; скорее ошибка)

## Статистика и операции

- Взнос, списанный со счёта — обычный расход (категория на выбор пользователя или отдельная «Первоначальный взнос» — не системная, на обсуждение)
- Платежи по графику — как у потребкредита (системная категория «Кредиты»)
- В статистике «переплата по процентам» — по сумме платежей минус погашенное тело; для ипотеки расчёт тот же, но точнее график

## Уведомления

- Без изменений триггера `credit_payment`: напоминание о ближайшем платеже по дате из `credit_payments`
- В шаблоне можно подставлять `credit_kind` / название «ипотека»

## Этапы внедрения

1. **MVP:** `credit_kind`, поля цена/взнос, `principal` = цена − взнос, график как у потребкредита (текущая формула процентов), без списания взноса
2. **Проценты:** дневная ставка между датами платежей; тесты по банковским выпискам
3. **Взнос со счёта:** операция при создании + отображение в карточке
4. **Однократная смена типа** у существующих кредитов
5. **Миграция данных:** существующие записи → `credit_kind = 'consumer'`

## Зависимости

- Модуль кредитов и график (v1.1: аннуитет, repair, редактирование сумм)

- [Кредитная карта](credit-card.md) — другая сущность, не путать с ипотекой

## Не в scope первой итерации

- Учёт залога / адреса объекта
- Досрочное погашение с пересчётом по правилам конкретного банка (частично есть как `early`)
- Несколько ипотек с объединением в один «портфель»
- Импорт ипотеки из Cubux как отдельный preset
