# Синхронизация с банками

## Зачем

Сейчас операции вносятся вручную или [импортом CSV/XLSX](../stage_06_import_export.md) (Cubux и др.). Автосинхронизация снижает рутину: новые транзакции появляются сами, остаётся разнести по категориям.

## Реальность РФ (2026)

- Универсального открытого API «все банки» для физлиц **нет**
- Варианты: официальные API отдельных банков (Тинькофф, Сбер, …) с OAuth; агрегаторы (с лицензией / платно); скрейпинг — хрупко и юридически сомнительно
- Для self-hosted продукта разумный путь: **плагины по банкам** + ручной импорт как fallback



## Архитектура (черновик)

```
connector (tinkoff | sber | …)
    → normalize → dedupe → import pipeline (как POST /import commit)
```

- Таблица `bank_connections`: user_id, bank_id, encrypted tokens, last_sync_at
- Периодический job (cron) или кнопка «Синхронизировать» в UI
- **Idempotency** — те же ключи, что [import idempotency](../docs/data-model.md); внешний id операции от банка



## Сопоставление со счетами

- Одна connection → один `account_id` (дебетовый `bank`)
- [Кредитные карты](credit-card.md) — тип счёта `credit_card`; документация — [ui-credit-cards.md](../docs/ui-credit-cards.md)



## Нормализация

Банк отдаёт: дата, сумма, описание, MCC, контрагент. Маппинг в:

- `type` expense/income (по знаку)
- `category_id` — правила + ML позже; сначала «Неразнесённое» + подсказки по истории
- `description` — из назначения платежа



## UI

- Настройки → «Подключения банков» (после OAuth redirect)
- Список неразнесённых операций (inbox) перед commit в основную книгу — или автocommit с категорией по умолчанию



## Связь с импортом

Переиспользовать `internal/importexport` где возможно: preview/commit, маппинг категорий, отчёт. Отличие — источник не файл, а API connector.
