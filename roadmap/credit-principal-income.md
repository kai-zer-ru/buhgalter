# Учёт дохода по кредиту в балансе

Задача v1.3.0 — опциональное отражение суммы потребительского кредита как дохода на счёте при создании.

Сейчас при создании кредита (не ипотека, не рассрочка) сумма займа **не влияет** на баланс счёта: учитываются только платежи по графику (ручные, автосписание, ретро-списания). Частый сценарий: кредит только что выдан, деньги поступили на дебетовый счёт — пользователь хочет сразу видеть актуальный баланс.

## Зачем

| Сейчас | Нужно |
|--------|--------|
| `principal_amount` только в графике кредита | Опционально — доход на `debit_account_id` в дату выдачи |
| Баланс счёта не меняется при создании | При включённом переключателе баланс += сумма кредита |
| Ретроактивный ввод — без операций по счёту | Переключатель заблокирован, если в графике есть платёж с датой в прошлом |

## Область действия

- Только **форма создания** кредита (`CreditForm`)
- Только тип **«Кредит»** (`credit_kind=consumer`, `interest_rate > 0`)
- **Не** ипотека, **не** рассрочка (`interest_rate = 0`)
- После сохранения — **не менять** опцию (read-only по факту `principal_transaction_id`)

## Поведение в UI

### Переключатель «Учитывать доход в балансе»

- Виден только при выборе типа «Кредит»
- По умолчанию **выключен**
- При включении и сохранении: операция `income` на `debit_account_id`, сумма = `principal_amount`, дата = `issue_date`, категория — системная «Кредиты» (доход)

### Блокировка при прошлых платежах в графике

Если в **графике платежей** есть хотя бы одна строка с датой **раньше начала текущих суток** (timezone пользователя):

- неважно, включён ли «Уже платил по графику» (`added_retroactively`);
- неважно, отмечен ли платёж «Списан со счёта» или только «Учтён при добавлении»;

→ переключатель **выключен**, **disabled**, подсказка: изменение баланса невозможно.

Если все даты платежей — сегодня или в будущем (даже при дате выдачи в прошлом), переключатель доступен.

## Фильтры и статистика

Системная категория дохода **«Кредиты»** (то же имя, что у расходной). При невыбранном фильтре «тип операции»:

- в селекте категорий (списки операций, статистика) — **«Кредиты (Доход)»** и **«Кредиты (Расход)»**
- в таблице «По категориям» на `/stats` — аналогично, в секциях доходов и расходов

Механизм — как у «Долги»: [`web/src/lib/category-label.ts`](../web/src/lib/category-label.ts).

## Модель данных

```sql
-- credits.principal_affects_balance INTEGER NOT NULL DEFAULT 0
-- credits.principal_transaction_id TEXT REFERENCES transactions(id)
```

Системная категория: `income`, имя «Кредиты», `is_system=1`.

## API

`POST /api/v1/credits`:

```json
{
  "principal_affects_balance": true
}
```

Валидация:

- только `consumer` + `interest_rate > 0`
- `principal_affects_balance=true` при наличии платежа в графике с датой &lt; сегодня → **400** (`ERR_CREDIT_PRINCIPAL_INCOME_PAST_PAYMENT`)

## Связь с другими опциями

- **[Списание ретро-платежей](credit-retroactive-debit.md)** — расходы по прошлым платежам; не пересекается с доходом по сумме кредита
- **[Ипотека](mortgage.md)** — `down_payment_affects_balance` (расход взноса); обратная семантика

## Не в scope

- Изменение опции после создания кредита
- Ипотека и рассрочка
- Отдельный счёт для дохода (всегда `debit_account_id`)
