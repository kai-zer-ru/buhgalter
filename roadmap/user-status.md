# Статус пользователя

Задача v1.3.0 — модерация самостоятельной регистрации и блокировка учётных записей.

Техническая документация: [docs/api/user-status.md](../docs/api/user-status.md).

## Зачем

Сейчас любой пользователь, зарегистрировавшийся при включённой регистрации, сразу получает полный доступ. Админ может только удалить пользователя. Нужен промежуточный статус «на проверке» и возможность заблокировать без удаления данных.

## Статусы

| Код API | RU (UI) | EN (UI) | Описание |
|---------|---------|---------|----------|
| `active` | Активен | Active | Полный доступ |
| `pending` | На проверке | Pending | Ожидает решения администратора |
| `banned` | Заблокирован | Banned | Доступ запрещён |

## Начальные значения

| Сценарий создания | Статус |
|-------------------|--------|
| Существующие пользователи (миграция) | `active` |
| Первый админ (setup) | `active` |
| Создание из админки | `active` |
| Самостоятельная регистрация | `pending` |

## Переходы в админке

Управление — меню «⋯» в списке пользователей (`/admin/users`).

| Текущий статус | Доступные действия |
|----------------|-------------------|
| `active` | Заблокировать |
| `banned` | Разблокировать |
| `pending` | Активировать, Заблокировать |

Ограничения:

- Администратор **не может** менять свой статус.
- Перевод в `pending` через админку **недоступен** (только при регистрации).

## Поведение при доступе

1. **Регистрация** — пользователь создаётся со статусом `pending`, сессия **не** выдаётся. UI показывает сообщение об ожидании активации и перенаправляет на страницу входа.
2. **Вход** — `pending` и `banned` получают `403` с понятным сообщением (не сырой код ошибки).
3. **Middleware** — все защищённые endpoints отклоняют неактивных пользователей (в т.ч. уже выданные сессии и API-токены при бане).
4. **Блокировка** — инвалидация всех сессий заблокированного пользователя.

## Сообщения об ошибках

На фронтенде пользователь видит только понятный текст на выбранном языке (RU/EN через i18n). Коды API (`USER_BANNED`, `USER_PENDING_MODERATION` и т.д.) — для машинной обработки, в UI не показываются.

## UI

- Колонка «Статус» в таблице пользователей.
- Действия в `RowActionsMenu` (см. [ui-row-actions.md](../docs/ui-row-actions.md)).
- Подтверждение перед блокировкой (как при удалении).
- **Плашка для админа** — как при сбросе пароля ([`AdminPasswordResetBanner`](../web/src/lib/components/AdminPasswordResetBanner.svelte)): список пользователей `pending`, кнопка перехода к модерации.
- **Попап модерации** на `/admin/users?moderate={id}` — активировать или заблокировать без поиска в таблице.

## Уведомления в мессенджеры

При успешной самостоятельной регистрации — исходящее уведомление **всем администраторам**, у которых:

- настроен хотя бы один канал (Telegram / MAX);
- включён триггер `user_registration` (только для админов, по аналогии с `password_reset`);
- **включена самостоятельная регистрация** в системных настройках (`registration_enabled`).

**Шаблон и переключатель триггера** в настройках уведомлений видны только администраторам и **скрываются**, если регистрация отключена.

Плейсхолдеры: `{login}`, `{display_name}`, `{registered_at}`, `{moderation_url}`.

`{moderation_url}` — ссылка на модерацию: `{external_url}/admin/users?moderate={user_id}`. Требует настроенный **внешний URL** в админке; иначе ссылка в сообщении не формируется (см. [telegram-max-buttons.md](telegram-max-buttons.md)).

Пример (RU): «Новая регистрация: {display_name} (@{login}), {registered_at}. Модерация: {moderation_url}»

## Модель данных

Колонка `users.status` (`TEXT`, `CHECK` на допустимые значения). Миграция `029_user_status.sql`.

## API

- `PUT /api/v1/admin/users/{id}/status` — смена статуса (тело: `{ "status": "active" | "banned" }`).
- Поле `status` в `User`, `AdminUser`, ответе регистрации.

См. [OpenAPI](../docs/api/openapi.yaml).
