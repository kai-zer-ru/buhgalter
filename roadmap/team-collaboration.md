# Командная работа

Не входит в v1. Черновик для обсуждения.

## Зачем

Сейчас каждый `user_id` полностью изолирован ([main_doc.md](../main_doc.md)): свои счета, категории, операции. Семейный или малого бизнеса учёт «одна база — несколько людей» требует общих данных и разграничения прав.

## Модели (на выбор)

**A. Общий «домохозяйство» (household)**

- Сущность `household` / `workspace`; пользователи — члены с ролями
- Счета, категории, операции привязаны к household, не к user
- Личные категории — опционально позже

**B. Совместные счета без полного слияния**

- У каждого свой набор данных, но счёт можно «расшарить» другому user (read-only / write)
- Меньше миграции, сложнее целостность статистики

**C. Организация + сотрудники**

- Как A, но роли admin / accountant / viewer; audit log уже есть ([stage_01](../stage_01_auth_admin.md))

Для личной бухгалтерии с семьёй чаще подходит **A**.

## Что меняется в данных

- `user_id` на операциях → `household_id` + `created_by_user_id` (кто внёс)
- Приглашения по ссылке / коду; принятие в household
- Категории и банки — общие или копия при вступлении (решить)
- API-токены — на уровне user или household?

## Миграция существующих установок

- Один пользователь → household из одного члена, данные перепривязать
- Без breaking change для solo: solo = household с одним участником

## UI

- Переключатель household (если пользователь в нескольких) — редкий кейс, можно отложить
- Индикация «кто создал» в списке операций
- Настройки: участники, роли, исключение

## Безопасность

- Нельзя читать чужой household без членства
- Удаление участника не удаляет историю операций
- Админ инстанса (system admin) ≠ admin household
