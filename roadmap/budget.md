# Бюджет

Не входит в текущие релизы. Черновик для обсуждения — не детальное ТЗ.

Сейчас в приложении есть **статистика** (`/stats`) — фактические доходы и расходы по категориям за выбранный период, и **периодические операции** — шаблоны будущих списаний. **Бюджет** — слой планирования поверх категорий: пользователь задаёт лимиты на период и видит прогресс «план vs факт», не дожидаясь конца месяца.

## Зачем отдельный модуль

| | Статистика (сейчас) | Периодические операции | Бюджет |
|---|---------------------|------------------------|--------|
| Вопрос | «Сколько потратил?» | «Что спишется само?» | «Сколько **можно** потратить?» |
| Период | любой, задним числом | бесконечный цикл | календарный месяц (MVP) |
| Гранулярность | категория / подкатегория | одна операция | категория, группа, общий лимит |
| Проактивность | нет | напоминание о списании | предупреждение при 80% / 100% |

Бюджет не заменяет статистику: факт берётся из тех же `transactions` (с теми же фильтрами, что `StatsByCategory` — без `exclude_from_stats` платежей кредита).

## Сценарии (MVP)

1. **Лимит на категорию** — «Продукты — 30 000 ₽ в месяц»; прогресс-бар на странице бюджета и бейдж в списке категорий.
2. **Общий лимит расходов** — «Все расходы — 150 000 ₽»; детализация по категориям опциональна.
3. **Обзор месяца** — переключатель месяцев, сводка: запланировано / потрачено / остаток.
4. **Уведомление** — при достижении порога (например 80% и 100%) в Telegram / MAX (новый триггер `budget_threshold`).

Не в MVP: бюджет на доход («план зарплаты»), мультивалютность, совместные бюджеты семьи, автоперенос остатка на следующий месяц.

## Модель данных (черновик)

**Таблица `budgets`** — шаблон лимита (не привязан к конкретному месяцу):

```sql
-- id TEXT PK
-- user_id TEXT NOT NULL REFERENCES users(id)
-- name TEXT NOT NULL                    -- «Продукты», «Всё», пользовательское имя
-- scope TEXT NOT NULL                   -- 'category' | 'subcategory' | 'all_expense' | 'all_income'
-- category_id TEXT REFERENCES categories(id)
-- subcategory_id TEXT REFERENCES subcategories(id)
-- amount INTEGER NOT NULL               -- лимит, копейки
-- period TEXT NOT NULL DEFAULT 'month'  -- MVP: только 'month'
-- account_id TEXT REFERENCES accounts(id)  -- NULL = все счета
-- rollover INTEGER NOT NULL DEFAULT 0   -- перенос остатка (post-MVP)
-- alert_at_percent INTEGER DEFAULT 80   -- порог уведомления, 0 = выкл
-- is_active INTEGER NOT NULL DEFAULT 1
-- created_at, updated_at TEXT
```

**Таблица `budget_periods`** — снимок на конкретный месяц (для истории и rollover):

```sql
-- id TEXT PK
-- budget_id TEXT NOT NULL REFERENCES budgets(id) ON DELETE CASCADE
-- period_start TEXT NOT NULL            -- первый день месяца в TZ пользователя, UTC в БД
-- planned_amount INTEGER NOT NULL       -- лимит на этот месяц (копия + корректировка)
-- rollover_amount INTEGER DEFAULT 0     -- перенесённый остаток с прошлого месяца
-- UNIQUE (budget_id, period_start)
```

Факт (`spent`) **не денормализуем** — считаем запросом к `transactions` за `[period_start, period_end)` с фильтрами бюджета. При необходимости — материализованный кеш в `budget_periods.spent_amount` + пересчёт по событию создания/изменения операции (как баланс счетов).

**Связь с категориями:**

- `scope = category` — сумма расходов с `category_id`
- `scope = subcategory` — только подкатегория
- `scope = all_expense` — все `type = expense` (опционально `account_id`)
- Системные категории («Комиссия», платежи кредита с `exclude_from_stats`) — **не входят** в бюджет по умолчанию; настройка «включать переводы / комиссии» — post-MVP

**Уникальность:** один активный бюджет на пару `(user_id, scope, category_id, subcategory_id, account_id)` — чтобы не дублировать «Продукты 30k» и «Продукты 25k».

## Расчёт факта

Переиспользовать логику `stats.ByCategory`:

- Период: `[начало месяца 00:00, начало след. месяца)` в `users.timezone`
- Только `kind = manual` и применённые `future` (как в статистике по умолчанию)
- Тип операции: `expense` для лимитов расходов
- Фильтр `account_id`, если задан в бюджете

**Прогноз до конца месяца (post-MVP):** `spent + sum(recurring_operations за оставшиеся дни по категории)` — переиспользовать данные периодических операций.

## API (черновик)

```
GET    /api/v1/budgets?month=2026-06
POST   /api/v1/budgets
PATCH  /api/v1/budgets/{id}
DELETE /api/v1/budgets/{id}

GET    /api/v1/budgets/summary?month=2026-06
       → список бюджетов с planned, spent, remaining, percent, status (ok|warning|exceeded)
```

Ответ `summary` — основной для UI; CRUD — для настроек.

## UI

### Страница `/budget` (новый пункт меню или вкладка в «Статистика»)

- Переключатель месяца (← июнь 2026 →)
- Карточки бюджетов: название, иконка категории, прогресс-бар, «18 400 / 30 000 ₽»
- Цвета: зелёный &lt; 80%, жёлтый 80–100%, красный &gt; 100%
- Кнопка «Добавить бюджет»
- Пустое состояние: «Задайте лимит на категорию, чтобы контролировать расходы»

### Форма бюджета

- Категория (селект только `expense`) или «Все расходы»
- Подкатегория — опционально
- Сумма, счёт (все / конкретный)
- Порог уведомления (%)
- Чекбокс «Активен»

### Главная (опционально, этап 2)

- Виджет «Бюджет месяца»: 2–3 категории с наибольшим % использования или одна строка «Расходы: 72% от плана»

### Статистика

- На вкладке «По категориям» — колонка «Бюджет» / «Остаток», если для категории задан лимит

## Уведомления

Новый триггер **`budget_threshold`**:

- Срабатывает при пересечении `alert_at_percent` и при 100% (один раз за период на бюджет)
- Текст: «Продукты: потрачено 24 000 из 30 000 ₽ (80%)»
- Проверка — в существующем worker уведомлений (раз в час или после операции — на обсуждение)
- Таблица `budget_alert_sent (budget_id, period_start, threshold_percent)` — дедупликация

Связь с [notifications-evolution.md](notifications-evolution.md) и планом «недостаточно денег перед списанием» — бюджет про **категории**, тот триггер про **баланс счёта**; не смешивать.

## Этапы внедрения

1. **MVP:** таблицы, CRUD, `GET summary`, страница `/budget`, лимит на категорию и «все расходы», месячный период
2. **Интеграция:** колонка в статистике, виджет на главной
3. **Уведомления:** триггер `budget_threshold`
4. **Прогноз:** учёт периодических операций до конца месяца
5. **Rollover:** перенос остатка, ручная корректировка `planned_amount` на месяц
6. **Расширения:** бюджет на доход, группы категорий («Дом» = ЖКХ + ремонт + мебель)
