# Сканер чеков

Планируется в **v1.5.0** ([ROADMAP](../ROADMAP.md#v150)). Web + Android.

## Зачем

Быстро создать расход из кассового чека: сумма, дата, магазин — без ручного ввода. Ориентир — **чеки РФ (54-ФЗ)**; зарубежные чеки не в приоритете.

## Уже пробовали (экспериментальная ветка) — не тянуть код вслепую

Локально была WIP-ветка (сейчас **`1.5.0`**, раньше называлась `1.4.0`, коммит вроде `check-scan-test`). Там сделан почти полный каркас: API, миграции магазинов/чеков, UI импорта, e2e с моками. **В бою оказалось нерабочим / хрупким** — ниже выводы, чтобы **не повторять мёртвые ходы**. ### Что было реализовано по архитектуре

| Кусок | Суть |
|-------|------|
| QR с камеры | Браузер (`html5-qrcode`) → `POST /receipts/parse` со строкой QR |
| Фото чека | Async job: `POST /receipts/scan-jobs` → опрос статуса; QR ищут **на сервере** (gozxing + go-qr, кропы низа, повороты EXIF, масштабы) |
| Состав чека | Цепочка ФНС (см. ниже) → preview позиций → `POST /receipts/import` (N расходов = N позиций, магазин `stores`, снимок JSON в БД) |
| Fallback | Если позиций нет — одна строка «Покупка по чеку» на всю сумму из QR |
| Env | `BUHGALTER_FNS_SESSION_ID` — sessionId из приложения ФНС «Проверка чеков» |
| Утилиты | `cmd/fns-probe`, `cmd/fns-fetch`, `cmd/receipt-services-probe` — опрос эндпоинтов |

Подробная «как было задумано»-дока жила на той ветке как `docs/receipt-scanner.md` (в текущем дереве её может не быть — смотреть в истории ветки `1.5.0`).

### Что по факту не работает / неприемлемо для продукта

1. **`proverkacheka.nalog.ru:9999`** (неофициальный GET с позициями)  
   С многих серверов (в т.ч. типичный VPS/Docker) — **timeout**. Иногда работает из «домашней» сети. Нельзя опираться как на основной канал self-host.

2. **`kkt-online.nalog.ru` (openapikkt)**  
   Официальный и доступный, но **только проверка подлинности** чека — **без списка товаров и магазина**. Сам по себе для импорта позиций бесполезен (максимум «чек настоящий» + fallback из QR).

3. **`irkkt-mobile.nalog.ru:8888` без `sessionId`**  
   Можно проверить факт чека; **позиций нет**. Полный чек только с сессией мобильного приложения.

4. **`BUHGALTER_FNS_SESSION_ID` (сессия приложения «Проверка чеков»)**  
   Технически даёт позиции, но:
   - сессия **короткоживущая** (~сутки);
   - добыча через mitm/перехват трафика, root/`shared_prefs`, или SMS API с `client_secret` из APK — **плохой UX для self-host**;
   - автообновления через `refresh_token` в прототипе **не было**;
   - неофициальный API — ФНС может отломать в любой момент.  
   **Не считать это нормальным способом поставки** для пользователей. Допустимо максимум как скрытый opt-in для технарей, не как «включи и пользуйся».

5. **Официальный `openapi.nalog.ru`**  
   Позиции возможны, но нужен **мастер-токен для ИП/ООО** — не подходит массово для физлиц self-host.

6. **Платные ОФД / proverkacheka.com**  
   Позиции да; нужен токен, лимиты freemium, чужой посредник. Имеет смысл как **явный BYO-ключ пользователя**, не как единственная зависимость сервера без настройки.

7. **Распознавание QR с «живого» фото на сервере**  
   Много проходов (кроп, порог, zoom) всё равно **часто не находит QR** на смазанных/кривых снимках. E2e зелёные за счёт **моков** job’а, не реальной камеры. Надёжнее читать QR **на устройстве** (камера + Barcode API / ML Kit) и на сервер слать уже строку `t=…&s=…`, а не надеяться на серверный decode фото.

8. **Fallback «Покупка по чеку»**  
   Формально импорт «работает», по смыслу почти бесполезен (нет магазина, нет позиций) — пользователь всё равно всё вбивает руками. Не продавать как успешный импорт.

### Что можно переиспользовать идеями (не копипастой)

- Модель «строка QR → preview → подтверждение → операции» — правильная.
- Dedup по `(fn, i, fp)` / фискальным полям — нужен.
- Магазин из чека → справочник — стыкуется с [merchants-tags.md](merchants-tags.md).
- Не хранить фото навсегда, если достаточно JSON чека — разумная приватность.
- Не тащить в продукт mitm/`sessionId` как основной путь.

## Целевое направление для v1.5.0

### A. QR на устройстве (главный путь для РФ)

1. Камера / scan в Android (и при возможности web) → строка QR 54-ФЗ.  
2. Сервер **не обязан** декодить QR из jpeg (серверный decode — только опциональный запасной, не KPI).  
3. Состав: **BYO** — токен пользователя к посреднику (например proverkacheka.com) **или** в будущем официальный путь, если появится нормальный для физлиц. Без ключа — честный UX: «есть реквизиты из QR, введите сумму/магазин вручную» или отказ, **без** притворного fallback.  
4. Prefill формы / таблица позиций → подтверждение → создание операций.  
5. Всегда ручное подтверждение.

### B. Разбор фото через нейросеть (ИИ)

Отдельный канал (как у ezBookkeeping): своя модель / свой ключ → prefill полей. **Не** замена ФНС по составу позиций; не чат с ИИ.

### C. Просто вложить фото

Прикрепить снимок к операции без разбора.

## Явно не делать снова

- Ставить `proverkacheka.nalog.ru:9999` основным источником «из коробки».
- Строить продукт на `BUHGALTER_FNS_SESSION_ID` / перехвате приложения ФНС.
- Обещать полный чек через один только `kkt-online`.
- Считать серверный QR-from-photo «готовым», пока нет стабильных e2e на реальных фото **без моков**.
- Молча создавать «Покупка по чеку» и называть это успехом.

## Scope v1.5.0 (кратко)

- Скан QR на Android (+ web, если камера тянет).
- Получение состава — осознанная схема ключей/посредника или честный ручной prefill из QR.
- Опционально ИИ по фото (настройка).
- Prefill: сумма, дата, **магазин**, описание; категория — [правила](category-rules-inbox.md) / вручную.
- Подтверждение перед сохранением.

## Не входит

- Пакет без просмотра.
- Чат с ИИ.
- Зарубежные фискальные форматы как MVP.
- «Бесплатные позиции ФНС из коробки без настроек» — после эксперимента это миф для типичного self-host.
