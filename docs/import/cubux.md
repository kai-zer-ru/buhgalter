# Формат Cubux (импорт / экспорт)

Buhgalter поддерживает импорт и экспорт операций в формате, совместимом с [Cubux](https://cubux.ru).

## Заголовки CSV

```
Тип,Дата,Сумма списания,Валюта списания,Счет списания,Сумма пополнения,Валюта назначения,Счет пополнения,Категория,Subcategory,Описание,Проект,Пользователь
```

Кодировка: **UTF-8 с BOM** (`\xEF\xBB\xBF` перед первой колонкой). Экспорт всегда добавляет BOM.

## Типы операций

| Cubux `Тип` | Действие в Buhgalter |
|-------------|----------------------|
| `Расходы` | expense: счёт = «Счет списания», сумма = «Сумма списания» |
| `Доходы` | income: счёт = «Счет пополнения», сумма = «Сумма пополнения» |
| `Перевод` | transfer: from = «Счет списания», to = «Счет пополнения» |

Важно: в файле Cubux `Перевод` — одна строка операции, но в БД создаются две связанные записи (`transfer` out/in с одним `transfer_group_id`).

## Формат значений

### Суммы

Импорт принимает Cubux-формат (`50.00_-₽`, `31024.46_-₽`) и обычные числа (`50.00`, `31024.46`).

Парсер извлекает число до символа `_` (или до первого нецифрового символа после дробной части), округляет до копеек.

Экспорт: десятичное число с двумя знаками после запятой, например `50.00`, `31024.46` — в колонках «Сумма списания» / «Сумма пополнения».

### Даты

`DD.MM.YYYY`, например `01.01.2025`.

### Валюта

В v1 игнорируется (одна валюта пользователя). При импорте не-RUB в лог пишется warning.

## Примеры строк (из `1.csv`)

```csv
Расходы,01.01.2025,50.00,RUB,Наличные,,,,Транспорт,Автобус,,,Максим Ковалёв
Доходы,12.01.2025,,,,1860.00,RUB,Яндекс,Прочие доходы,Авито,,,Максим Ковалёв
Перевод,06.01.2025,3680.00,RUB,Яндекс,3680.00,RUB,Кредитка,Перевод,,,,Максим Ковалёв
```

## Импорт в Buhgalter

API:

- `POST /api/v1/import/preview` — dry-run (multipart: `file`, `preset=cubux`, `deduplicate`)
- `POST /api/v1/import` — commit (`confirm=true`, опционально `account_map`, `category_map`, `subcategory_map`, `auto_subcategory`, заголовок `Idempotency-Key`)
- `POST /api/v1/import/jobs` — фоновый commit (возвращает `job_id`, статус `queued`)
- `GET /api/v1/import/jobs/{id}` — статус фонового импорта (`queued|running|done|failed`) и итоговый `report`

### Async job: прогресс и лог

`GET /api/v1/import/jobs/{id}` в поле `report` возвращает промежуточные данные уже во время выполнения:

- `total_rows` — строк в файле;
- `processed_rows` — сколько уже обработано;
- `created_transactions`, `skipped_duplicates`, `errors` — текущая сводка;
- `logs[]` — построчный лог обработки (импорт, skip duplicate, mapping error, error).

Фронтенд `/import` на основе этого показывает прогресс-бар, ETA и спойлер «Полный лог».

### Устойчивость фонового импорта

- При рестарте сервера старые job в `queued/running` автоматически переводятся в `failed` с причиной `import interrupted: server restarted`.
- На коротких блокировках SQLite (`SQLITE_BUSY` / `database is locked`) применяется retry с backoff, чтобы импорт не терял пачки строк.

UI: `/import` — wizard:
загрузка → настройки → сопоставление счетов (уникальные) → сопоставление категорий (уникальные) →
подкатегории (если отключен авто-подбор) → превью-сводка → подтверждение через модалку.

### Автосоздание

- **Счета** — шаг wizard «Сопоставление счетов»: все уникальные имена из файла; точное совпадение (без учёта регистра) → привязка к существующему, иначе → создать новый. Имена известных банков (ВТБ, Сбер, ВБ, Яндекс и т.д.) предлагаются как банковские счета с выбранным банком; остальные — как наличные. Тип (`cash`, `bank`, `credit_card`) и банк можно изменить перед импортом; для `credit_card` дополнительно указывается лимит. См. [ui-credit-cards.md](../ui-credit-cards.md).
- **Категории** — шаг «Сопоставление категорий»: все уникальные пары `{category, type}`; fuzzy match без учёта регистра; при отсутствии — создаётся expense/income категория.
- Если категория уже существует (гонка или параллельное создание), импорт переиспользует существующую категорию и не падает с `category name already exists`.
- **Подкатегории**:
  - если `auto_subcategory=true` (по умолчанию) — подбираются автоматически внутри выбранной категории; при отсутствии создаются;
  - если `auto_subcategory=false` — появляется шаг «Сопоставление подкатегорий» (уникальные), можно выбрать existing/create вручную.

### Дедупликация

Хеш: `date|amount|account|category|type` (имена счёта и категории, дата `YYYY-MM-DD`, сумма в копейках).

При `deduplicate=true` пропускаются дубликаты внутри файла и совпадения с уже существующими операциями в БД.

## Экспорт

`GET /api/v1/export?from=2025-01-01&to=2025-12-31&account_id=uuid`

- Content-Type: `text/csv; charset=utf-8`
- Content-Disposition: `attachment; filename="buhgalter_export_YYYY.csv"`
- Колонка `Пользователь` = `users.display_name`

## Известные edge cases (`1.csv`)

| Проблема | Поведение |
|----------|-----------|
| Дублирующиеся строки (стр. 2–3) | Отсекаются дедупликацией |
| Мусорные / неполные строки | Ошибка в отчёте, строка пропускается |
| Счёт «Яндекс» | Создаётся как банковский (Яндекс Банк), если нет существующего счёта с таким именем |

## XLSX

Поддерживается импорт первого листа `.xlsx` (библиотека excelize). Структура колонок — как в CSV Cubux.
